# Alerts & Notifications

## User Interaction

Alert dashboard shows real-time policy violations across build/deploy/runtime with filters for severity, policy, cluster, deployment, lifecycle stage. Alert details include violation specifics, evidence, deployment context, and remediation guidance. Lifecycle states: active, resolved, attempted with timestamps.

Notification integrations push alerts to Slack, PagerDuty, JIRA, email, webhooks with customizable policy-to-notifier mappings. Audit notifications send compliance events to separate channels. Vulnerability reports deliver scheduled CVE data via email/Slack.

## Code Locations

**Alert Management**: `central/alert/datastore/` stores and queries alerts, `central/alert/service/` provides gRPC API, `central/detection/lifecycle/` manages state transitions.

**Notifiers**: `pkg/notifiers/` has core interfaces and plugin system. Individual plugins at `pkg/notifiers/**/` implement 13+ integrations (Slack, PagerDuty, JIRA, Email, Splunk, Syslog, Generic Webhook). `central/notifiers/` has gRPC API, `central/notifier/processor/` routes alerts to integrations.

**Reports**: `central/reports/` manages scheduled vulnerability/compliance reports, `central/reports/scheduler/` runs cron jobs, `central/reports/common/` formats CSV/PDF/email.

**Database**: `central/alert/datastore/internal/store/postgres/` implements PostgreSQL schema with SAC.

## Data Model

**Alert**: ID, policy reference, lifecycle stage (BUILD_TIME/DEPLOY_TIME/RUNTIME), entity (deployment/image/resource), cluster/namespace, state (ACTIVE/RESOLVED/ATTEMPTED), enforcement action taken, violations list, timestamps (first occurred, last updated, resolved), process violation data, common entity info.

**ListAlert**: Lightweight subset for list views, optimized for dashboard performance.

**Notifier**: ID, name, type (SLACK/PAGERDUTY/JIRA/EMAIL/etc), config (webhook URL, API keys, etc), label key/value for filtering, enabled state.

**Type-specific configs**: Slack (webhook URL, channel), PagerDuty (integration key), JIRA (project key, issue type, API token), Email (SMTP server, from, recipients), Generic Webhook (URL, headers, custom body template), Splunk (HEC endpoint, token), Syslog (server, port, protocol).

**ReportConfiguration**: Name, type (vulnerability/compliance), schedule (cron), notifiers list, scope (cluster/namespace filters), CVE filters (severity, fixability).

## Pipeline

### Alert Generation

Policy violation detected by `central/detection/` which determines enforcement action. `central/alert/datastore/` creates alert with policy ID, violated entity, violation messages, enforcement action, lifecycle stage. Deduplication checks for existing active alert with same policy+entity, updates timestamp instead of creating duplicate. State transitions: new=ACTIVE, deployment deleted=RESOLVED (auto), user marks resolved=RESOLVED, enforcement failed=ATTEMPTED. Notification processor receives alert from `central/notifier/processor/`.

### Notification Routing

System queries enabled notifiers, filters by policy annotation matching notifier labels and configured scope. If no policy annotation, uses all enabled notifiers. `pkg/notifiers/` formats alert for each type: Slack (rich message with buttons), JIRA (issue title/description), Email (HTML with tables), Webhook (JSON payload). Plugin-specific logic sends: Slack POST to webhook, PagerDuty POST to Events API v2, JIRA REST API create issue, Email via SMTP, Splunk HEC POST, Syslog UDP/TCP/TLS. Errors handled with retry/exponential backoff for transient failures, logging for persistent errors, queueing/batching for rate limiting.

### Reports

`central/reports/scheduler/` runs cron jobs, checks for due reports. Vulnerability report queries image CVEs from datastore, filters by severity/fixability/cluster/namespace, aggregates by deployment/image/CVE. Compliance report runs scan (see compliance.md), collects check results for standards. `central/reports/common/` formats as CSV or PDF with summary statistics, tables, charts. Delivery sends to configured notifiers: Email attaches PDF or CSV, Slack uploads file, Webhook POSTs report data.

### Audit Notifications

Audit events (user login/logout, role change, configuration update) generated by `central/audit/`. Only notifiers configured for audit events receive them, separate from alert notifications. Format includes event type, user, timestamp, affected resource. Sent to SIEM systems (Splunk, Syslog).

## Configuration

**Environment Variables**: `ROX_NOTIFIERS_ENABLED` (enable notifications, default: true), `ROX_ALERT_RETENTION` (days to retain, default: 30), `ROX_NOTIFIER_TIMEOUT` (send timeout seconds, default: 10), `ROX_EMAIL_NOTIFIER_ENABLED` (enable email, default: true).

**Policy Notifier Mapping**: Policy annotation `notifier_labels: "critical-alerts"` sends to notifiers tagged with `labelDefault: "critical-alerts"`. If no annotation, all enabled notifiers receive alerts.

**Report Scheduling**: Daily/weekly/cron expression, CVE filters (Critical/High severity, fixable only), scope (all clusters or specific namespaces), notifiers (Email, Slack, webhook).

## Known Issues

**Performance**: 1000+ alerts/hour overwhelm UI and database. Policy changes trigger mass re-evaluation and alert flood. Vulnerability reports for 10,000+ images take minutes.

**Reliability**: Transient network errors lose notifications. Slack/PagerDuty rate limits drop notifications. SMTP issues or spam filters block alerts.

**Behavior**: Similar violations create separate alerts (different container in same pod). Deleted deployments auto-resolve alerts which may not be desired. Slight delay between alert creation and notification send.

**Limitations**: Each alert sends separate notification (no aggregation). Generic webhook templates have limited customization. JIRA issues not synced back. No notification history visible. Very large reports (100+ MB) may fail to send.

Workarounds: Use policy scope to reduce volume, configure notifier labels for critical alerts only, batch notifications via scheduled reports instead of real-time, use generic webhook for custom aggregation, implement external notification gateway for advanced routing, increase `ROX_ALERT_RETENTION` and periodically export to external storage, use Splunk/Syslog for durable logging, create multiple notifier instances for different channels.
