## Prototype Summary

This branch (`dashrews/prototype/finding-as-advisory`) implements **Variant 1: Finding = Advisory**.

Each advisory match from scanner is stored as its own finding row. CVE name is a grouping key resolved at query time via `GROUP BY`. There is no dedup logic — every advisory flows through as a legitimate, separate finding.

### Key Changes

**Scanner** (`pkg/scannerv4/mappers/`)
- Resolves CVE names and populates `CveName`, `AdvisoryId`, `SourceName` on each Vulnerability output
- Removes `dedupeVulns()` — all advisories flow through
- `updaterDisplayName()` maps updater strings to human-readable source names

**Storage** (3 new tables, created via `gen.go` + `pg-table-bindings-wrapper`)
- `image_scan_v2` — scan metadata (scanner version, bundle version, data sources)
- `scan_components` — packages found during scan (now includes arch, module, source_package, cpe, kind fields)
- `scan_findings` — one row per advisory per component per image
- Schema generated by `go generate` with `--schema-only` flag
- Type mappings in `tools/generate-helpers/pg-table-bindings/list.go`

**Central** (`pkg/scanners/scannerv4/convert.go`)
- Simplified to 1:1 passthrough using explicit `cve_name` from scanner

**Ingestion** (`pkg/scanners/scannerv4/scannerv4.go` + `central/scandata/ingestion/`)
- Callback pattern (`ScanIngestFunc`) avoids `pkg/` importing `central/`
- Callback registered in `central/main.go` at startup
- Every scan automatically populates prototype tables alongside existing v2 tables
- Populates `FirstImageOccurrence` and `FirstSystemOccurrence` timestamps at scan time

**REST API** (`central/scandata/api/`)
- `GET /v1/scandata/cves` — CVE list with GROUP BY aggregation, sorting, and pagination
- `GET /v1/scandata/cves/{cveName}` — CVE detail with advisories, components, images
- `GET /v1/scandata/advisories` — advisory list across all images with sorting and pagination
- `GET /v1/scandata/components` — component list with CVE counts
- `GET /v1/scandata/components/{name}` — component detail with versions and CVE breakdown
- `GET /v1/scandata/deployments` — deployment list with CVE counts via JOIN, sorting and pagination
- `GET /v1/scandata/deployments/{id}` — deployment detail with per-image CVE summary
- `GET /v1/scandata/images` — image list with sorting and pagination
- `GET /v1/scandata/images/{imageId}` — image detail with components and per-component CVEs
- `GET /v1/scandata/images/{imageId}/findings` — raw findings for an image

**CSV Export** (`central/scandata/export/`)
- Grouped-by-CVE format (default) and raw-findings format
- Backend implemented, UI export button not yet wired

**UI** (`ui/apps/platform/src/Containers/Vulnerabilities/proto/`)
- Accessible via sidebar: Vulnerability Management → Vuln Management V5
- Five tabs via `ProtoNav`: CVEs, Advisories, Components, Images, Deployments
- **CVE list**: simplified 8-column table (CVE, Top Severity, Top CVSS, Images, Fixable, First Seen, Published, EPSS %) with tooltips explaining aggregation. Sortable and paginated. EPSS displayed as percentage (7.9%) not raw float.
- **CVE detail**: description from highest-severity advisory in header, advisory table with links/descriptions/fix versions (includes NVD links for OS advisories like Alpine/RHEL), affected components (grouped by name+version with drill-down links), affected images (expandable showing per-component advisory IDs). Three layout variants (flow, tabs, collapsible) selectable via toggle. First seen timestamps populated from scan ingestion.
- **Advisory list**: all distinct advisories with external links (GHSA→GitHub, GO→pkg.go.dev, RHSA→Red Hat, CVE→NVD), CVE association, severity, description, component count. Sortable and paginated.
- **Component list**: components with version counts, CVE counts (distinct, not per-finding), severity breakdown. Sortable and paginated.
- **Component detail**: versions of component with CVE lists per version, images tab showing which images contain this component. Links to CVE detail. Severity counts fixed to COUNT(DISTINCT cve_name).
- **Image list**: images with OS, cluster, namespace, CVE counts. Sortable and paginated.
- **Deployment list**: deployments with cluster, namespace, image count, CVE count, top severity. Sortable and paginated.
- **Deployment detail**: images in deployment with per-image CVE summary
- **Image detail**: scan info, CVE severity summary badges, expandable components table with per-component CVE list and advisory IDs
- Image links enriched with name/UUID from `images_v2` table

### Architecture Decisions Made During Implementation

| Decision | Detail |
|----------|--------|
| REST API over GraphQL | Simpler for prototype — no schema registration, resolvers, or type matching needed. GraphQL resolvers still exist but are unused by the UI. |
| Callback pattern for ingestion | `pkg/scannerv4/` can't import `central/` (roxvet violation). `ScanIngestFunc` callback set from `central/main.go` at startup. |
| Schema via gen.go | Hand-written GORM schemas caused column name mismatches. Using `pg-table-bindings-wrapper --schema-only` generates correct schemas. |
| No FK to images_v2 | Ingestion fires inside `GetScan()` before image is stored in `images_v2`. FK dropped for prototype; image names enriched at query time instead. |
| Advisory dedup in API, not storage | Same advisory appears on multiple images as separate rows. API deduplicates advisories by ID when building CVE detail response. Per-image component expansion deduplicates by name+version and collects advisory IDs. |
| Query params for component API | Component names can contain slashes (e.g., `google.golang.org/grpc`), breaking path-based routing. Use `?name=...` query param instead. |
| Scanner version population | Scanner V4 service populates `scanner_version` (from `scanner/internal/version.Version`) and `bundle_version` (last vulnerability DB update in RFC3339 format) in `VulnerabilityReport`. These flow through ingestion to `image_scan_v2` table and are returned by `/v1/scandata/images/{id}` API. |

### Future Capability: Scan History and Diff Detection

**The `image_scan_v2` layer enables scan history and change detection** — a capability NOT in this prototype but designed into the schema. By introducing a scan entity between image and findings, the data model supports:

- **Multi-scan retention**: Keep N scans per image with TTL-based cleanup (currently we delete old scan on rescan)
- **"What changed?" queries**: `SELECT * FROM scan_findings WHERE scan_id IN (current_scan, previous_scan)` shows exactly which CVEs appeared or disappeared between scans
- **Scan metadata history**: Track which scanner version and vuln DB bundle produced each scan result
- **Audit trail**: Answer "when did this CVE first appear?" and "what scan last saw this CVE?"

Enabling this requires only **ingestion logic changes** (stop deleting old scans, add TTL cleanup job) — **no schema migration**. The prototype uses latest-scan-only to keep it simple, but the foundation is already there.

### What's NOT in this prototype

- Migration of existing data (new tables populated by new scans only)
- Dual-write / feature flag infrastructure
- Policy engine changes
- Materialized summary table (Approach C — only if GROUP BY is slow)
- Exception/deferral system changes
- Search framework integration (raw SQL queries used instead)
- CSV export UI button (backend exists)
- Branch 2 (finding-as-cve variant with JSONB advisories)

### Known Issue: Advisory ID Collisions Across Sources

When multiple data sources use the CVE ID itself as the advisory ID (e.g., Alpine SecDB reports `CVE-2026-27135` and Red Hat VEX also reports `CVE-2026-27135`), ingestion deduplicates by advisory ID and only keeps the first one processed. This means one source's finding is silently dropped.

In practice this is acceptable when an RHSA is also present (the RHSA is the actionable Red Hat advisory, making the VEX CVE-as-advisory redundant). However, if no RHSA exists, the dropped finding means we lose visibility into one data source's assessment (severity, CVSS, description may differ between sources).

**Decision needed for production:** Should advisory ID uniqueness be scoped per-source (i.e., `(advisory_id, source_name)` as the dedup key instead of just `advisory_id`)? This would preserve both findings but increase row count for OS-level CVEs where every source uses the CVE ID as the advisory.

### Definition of CVE Uniqueness

A CVE instance is unique by: **(cve_name, component_name, component_version, component_source, image_id)**.

Multiple advisories (GHSA, GO vulndb, RHSA) for the same CVE on the same component in the same image are separate findings — not duplicates. For OS-level vulnerabilities (Alpine, Debian, RHEL), the advisory ID IS the CVE name because those sources don't have a separate advisory system.
